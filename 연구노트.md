## 1. 배경

### 연구 동기

소형 언어모델(SLM, 1B~8B 파라미터)은 엣지 환경·온프레미스 배포에서 비용·프라이버시 이점이 있지만, 대형 모델 대비 지시 따르기(instruction-following)와 구조화 출력의 안정성이 낮다는 실무적 문제가 있다. 특히 JSON 추출처럼 포맷이 엄격히 요구되는 작업에서 어느 조건에서 "붕괴(collapse)"가 발생하는지 정량적으로 알려진 데이터가 부족했다.

### 핵심 질문

1. **기법 효과**: 프롬프팅 기법(zero-shot/few-shot/CoT 등)에 따라 구조화 출력의 안정성이 얼마나 달라지는가?
2. **문서 길이 효과**: 입력 문서의 토큰 수(token budget)가 어느 구간에서 안정성 저하를 유발하는가?

### 태스크 설계

한국어 공공기관 문서에서 7개 필드(기관명·문서번호·작성일·제목·핵심내용·담당자·처리기한)를 JSON으로 추출하는 과제를 선택했다. 선택 이유:

- 정답이 명확하게 정의되는 구조화 필드 존재 (문서번호, 날짜 등)
- 요약 능력이 필요한 자유 형식 필드 혼재 (핵심내용)
- 실무 적용 가능한 도메인

## 2. 실험 설계

### 독립변수

| 변수 | 수준 |
|------|------|
| **모델** | qwen2.5:1.5b, qwen2.5:3b, qwen2.5:7b |
| **프롬프팅 기법** | zero_shot, few_shot, cot, structured, xml, json_schema |
| **token budget** | 200, 500, 1000, 2000, 4000 (토큰) |

셀 수: 3 × 6 × 5 = **90개 셀**, 셀당 50 샘플 → 총 **4,500회 추론**

### 종속변수

| 메트릭 | 설명 |
|--------|------|
| `is_valid_json` | JSON 파싱 성공 여부 (구조적 유효성) |
| `schema_valid` | 7개 필드 키 모두 존재 여부 (스키마 준수) |
| `semantic_score` | 7개 필드 값의 의미적 정확도 평균 (0~1) |
| `score_{필드명}` | 필드별 개별 점수 |

### 실험 환경

- **추론 엔진**: Ollama (temperature=0.0, num_ctx=8192)
- **데이터 생성 모델**: gpt-5.4-mini-2026-03-17
- **채점 보조 모델**: gpt-4o-mini (담당자·제목·핵심내용 3개 필드)
- **실행 기기**: Windows 11, GPU 로컬 환경

## 3. 진행 내용

### 3.1 데이터 생성 (`scripts/generate_data.py`)

GPT API로 한국어 공공기관 스타일 문서와 정답 JSON 쌍을 합성했다.

- **문서 유형**: 공문·보고서·규정집·회의결과·사업계획서 중 무작위 선택
- **기관 유형**: 중앙부처·지자체·공공기관·교육청 중 무작위 선택
- **도메인**: 환경·복지·교통·예산·인사·시설관리 중 무작위 선택
- **토큰 수 제어**: Qwen 기준 1토큰 ≈ 2 한국어 글자로 환산해 목표 글자 수 지정
- **실제 토큰 수 검증**: ollama.generate(num_predict=0)로 Qwen 토크나이저 기준 실제 토큰 수 측정 후 기록

출력: `data/raw/budget_{N}/raw_{N}_{i}.json` (budget별 50개 = 총 250개 문서)

**토큰 수 분포 (budget별 실측값)**

| token_budget | min | max | mean | std | ±30% 초과율 |
|---:|---:|---:|---:|---:|---:|
| 200  | 212 | 290  | 255  | 15  | 32% |
| 500  | 504 | 874  | 648  | 88  | 46% |
| 1000 | 715 | 1731 | 1231 | 213 | 42% |
| 2000 | 1224 | 2908 | 2177 | 428 | 20% |
| 4000 | 2196 | 4096 | 3726 | 551 | 10% |

인접 budget 구간 간 실제 토큰 수 중첩이 존재한다(500 max > 1000 min 등). collapse_threshold는 "붕괴 확정점"이 아닌 "안정성 저하 관찰 시점"으로 해석해야 한다.

### 3.2 실험 실행 (`scripts/run_experiment.py`)

90개 셀을 순회하며 Ollama 모델 추론을 실행했다.

```
for model in [qwen2.5:1.5b, 3b, 7b]:
    for technique in [zero_shot, few_shot, cot, structured, xml, json_schema]:
        for budget in [200, 500, 1000, 2000, 4000]:
            for idx in range(50):
                → 프롬프트 조립 → 추론 → result_{model}_{technique}_{budget}_{idx}.json
```

출력: `experiments/{model-slug}/{technique}/result_*.json` (4,500개 파일)  
저장 항목: 모델명, 기법, token_budget, 입력 문서, 정답, raw_output, 지연 시간, 토큰 수

실험은 3회에 걸쳐 점진적으로 완성됐다(중간 중단 후 재시작 시 기존 파일 스킵 처리).

| 실행 | 일시 | runs_per_cell | 신규 생성 | 스킵 |
|------|------|---:|---:|---:|
| 1차 | 2026-05-18 14:58 | 20 | 1,708 | 92 |
| 2차 | 2026-05-18 17:21 | 50 | 2,160 | 1,800 |
| 3차 | 2026-05-19 08:38 | 50 | 540 | 3,960 |

### 3.3 평가 (`scripts/evaluate.py`)

채점은 v1 → v2로 개선됐다. v1에서 발견된 세 가지 구조적 문제를 해결했다.

**v1의 문제점**

| 문제 | 내용 |
|------|------|
| 날짜 포맷 오채점 | 값은 맞고 포맷만 다른 경우(한국어 날짜)를 GPT가 불일치로 처리 |
| 필드 뭉개기 | 7필드를 단일 점수로 평가해 어느 필드가 실패했는지 알 수 없었음 |
| GPT rubric 모호 | "절반 정도 일치=0.5" 기준이 불명확해 GPT 재량 컸음 |

**v2 채점 기준**

*코드 처리 필드 (4개)*

| 필드 | 채점 방식 |
|------|-----------|
| 문서번호 | 공백 정규화 후 exact match. 1.0 / 0.0 |
| 작성일·처리기한 | 날짜 값+포맷 일치 1.0 / 값은 맞고 포맷만 다름(한국어) 0.8 / 오류 0.0 |
| 기관명 | 완전 일치 1.0 / 기관명만 일치(부서명 누락) 0.5 / 기타 0.0 |

*GPT 처리 필드 (3개, 1회 호출로 묶음)*

| 필드 | 채점 방식 |
|------|-----------|
| 담당자 | 이름+직책 모두 일치 1.0 / 이름만 일치 0.5 / 그 외 0.0 |
| 제목 | 핵심 단어 모두 포함 1.0 / 일부 누락·변형 0.5 / 전혀 다름 0.0 |
| 핵심내용 | 정보 항목 모두 포함·간결 1.0 / 일부 포함 또는 원문 복사 0.5 / 핵심 없음 0.0 |

```
semantic_score = mean([기관명, 문서번호, 작성일, 제목, 핵심내용, 담당자, 처리기한])
```

채점 소요 시간: 약 86분 (2026-05-19 12:47~14:13)

**채점 실행 중 발생한 버그 (v2)**

| 버그 | 발생 시점 | 원인 | 수정 |
|------|-----------|------|------|
| `_extract_json`이 list 반환 | 407개 처리 후 크래시 | JSON 배열로 파싱된 경우 dict 검증 없음 | `isinstance(obj, dict)` 체크 추가 |

| 한국어 날짜 파싱 실패 | 테스트 실행 후 점수 전부 0.0 | `dateutil`이 `2024년 5월 18일` 미지원 | 정규식으로 직접 파싱하는 `_parse_date` 작성 |

## 4. 주요 결과

### 4.1 모델별 결과

| 모델 | is_valid_json | schema_valid | semantic_score |
|------|---:|---:|---:|
| qwen2.5:1.5b | 0.9867 | 0.8187 | 0.8013 |
| qwen2.5:3b   | 1.0000 | 0.9953 | 0.8840 |
| qwen2.5:7b   | 0.9960 | 0.9900 | 0.9151 |

qwen2.5 시리즈 내에서 파라미터 수가 많을수록 세 메트릭 모두 높게 관찰됐다. 단, 세 모델은 파라미터 수 외 학습 데이터·RLHF 방식이 통제되지 않았으므로 "모델 크기 효과"로 단정할 수 없다.

### 4.2 기법별 결과

| 기법 | is_valid_json | schema_valid | semantic_score |
|------|---:|---:|---:|
| few_shot    | 1.0000 | 0.9893 | 0.9249 |
| cot         | 1.0000 | 0.9987 | 0.9023 |
| json_schema | 0.9973 | 0.9413 | 0.8833 |
| zero_shot   | 0.9827 | 0.8973 | 0.8610 |
| xml         | 0.9893 | 0.7867 | 0.8194 |
| structured  | 0.9960 | 0.9947 | 0.8097 |

few_shot이 semantic_score 최고(0.9249). structured가 semantic_score 최저(0.8097)이지만 schema_valid는 높다(0.9947).

### 4.3 주요 발견 사항

**1. few_shot이 날짜·담당자 포맷 준수율 최고**  
`score_작성일 = 1.0`, `score_담당자 = 1.0`으로 전 기법 중 유일하게 완벽하다. 예시가 날짜 포맷(YYYY-MM-DD)과 직책 표기 방식을 직접 보여주는 효과로 해석된다.

**2. structured 기법의 기관명 부서명 누락**  
structured 기법에서 모델과 무관하게 기관명 부서명 누락률이 40% 이상이다.

| 기법 | 1.5b | 3b | 7b |
|------|---:|---:|---:|
| structured | 45.4% | 40.0% | 41.1% |
| few_shot   | 0.0%  | 0.0%  | 0.0%  |
| cot        | 10.0% | 0.4%  | 2.0%  |

`Step 1. 발신 기관 또는 담당 부서를 확인한다` 지시가 모델로 하여금 기관명만 추출하게 유도하는 것으로 추정된다.

**3. qwen2.5:3b + structured 조합의 문서번호 빈값 이상**  
qwen2.5:3b에서 structured 기법 시 문서번호 빈값 출력이 47.2%다. 1.5b(2.0%), 7b(0.8%)와 뚜렷이 다르며 카이제곱 검정으로 통계적 유의성 확인됐다(Bonferroni 보정 후 p<0.001).

**4. semantic_score 천장 효과**  
전체 4,500건의 56.8%(2,557건)가 semantic_score=0.9286이라는 단일 값에 집중된다. 구조: 핵심내용 점수가 0.40~0.50에 수렴하고 나머지 6개 필드가 대부분 1.0이라 (6×1.0 + 0.5)/7 = 0.9286에 집중된다. v3 재채점(핵심내용 루브릭 개선)으로 해결 예정.

**5. xml 기법의 역방향 패턴 (qwen2.5:1.5b)**  
1.5b + xml에서 schema_pattern=recovery. budget_200에서 schema_valid=0.00이고 budget_4000에서 0.62로 상승한다. 짧은 문서에서 핵심내용 키를 생략하는 현상에 기인한다.

**6. few_shot의 프롬프트 오버헤드**  
few_shot 기법은 고정 예시 2개로 인해 다른 기법 대비 평균 712 토큰의 오버헤드가 발생한다. few_shot의 성능 우위가 예시 효과에 의한 것인지 더 긴 컨텍스트 효과에 의한 것인지 분리하기 어렵다.

### 4.4 통계 검정 요약

- **검정 방법**: 이진 메트릭(카이제곱), 연속 메트릭(Mann-Whitney U), Bonferroni 보정 (α=0.00333, 15쌍)
- `schema_valid`: 15쌍 중 12쌍 유의. xml·zero_shot이 cot·few_shot·structured와 뚜렷하게 다름.
- `semantic_score`: 15쌍 중 13쌍 유의. structured가 cot(r=−0.605, large), few_shot(r=−0.755, large)과 가장 큰 차이.
- `is_valid_json`: 대부분 쌍에서 유의차 없음. JSON 파싱 성공률은 기법보다 모델에 더 민감.

## 6. 한계 및 후속 과제

### 한계

1. **재현성**: temperature=0.0은 재현성에 근접한 조건이지 완전한 보장이 아니다. GPU 부동소수점 누적 오차, llama.cpp 연산 순서 등으로 동일 입력에서도 극소수 출력 차이가 발생할 수 있다.
2. **구간 중첩**: 인접 budget 구간 간 실제 토큰 수가 중첩돼 경계값을 확정할 수 없다.
3. **few_shot 오버헤드**: 기법 간 총 프롬프트 토큰 수가 통제되지 않아 few_shot 성능 원인 분리가 어렵다.
4. **핵심내용 변별력**: v2 루브릭에서 핵심내용 점수 85.4%가 0.5에 집중돼 기법 간 차이를 포착하기 어렵다.
5. **xml 필터링 편향**: xml 기법은 schema_valid=False 비율이 21.3%로 타 기법 대비 높아, semantic_score 직접 비교 시 편향 발생 (schema_valid=True 필터링 시 +0.035 상승).

### 후속 과제

- **v3 재채점**: 핵심내용 루브릭 개선 후 재채점하여 천장 효과 해소
- **few_shot 오버헤드 통제 실험**: 컨텍스트 길이를 통제한 조건에서 예시 효과 분리
