## Ground Truth 검증

budget별 각 10개 샘플(index 0~9), 총 50개 샘플에 대해 GT JSON 필드가 실제 문서 내용과 일치하는지 수동 검토했다.

### 검증 방법

- 구조화 필드 6개(기관명·문서번호·작성일·제목·담당자·처리기한): GT 값이 문서 본문에 존재하는지 확인
  - 작성일·처리기한은 ISO 날짜를 한국어 날짜로 역변환(`2024-05-18` → `2024년 5월 18일`)하여 문서 내 존재 여부 확인
- 핵심내용: GT 값이 문서 본문의 직접 복사가 아닌 요약인지 확인

### 검증 결과

| 항목 | 결과 |
|------|------|
| 구조화 필드 오류 (기관명·문서번호·작성일·제목·담당자·처리기한) | 0 / 50 (0%) |
| 핵심내용 전체 복사 | 0 / 50 (0%) |
| 핵심내용 문장 단위 직접 인용 | 1 / 50 (2%) — raw_500_2 |

구조화 필드는 전 샘플에서 문서 내용과 일치했다. 핵심내용은 GPT가 실제 요약을 생성했고, 1건(2%)에서 문장 하나가 원문과 동일했다.

### 핵심내용 필드 한계

핵심내용은 GPT가 생성한 요약이므로, 동일 문서에 대해 다른 표현으로 작성된 요약도 정보적으로 동등할 수 있다. 예를 들어 "예산 삭감 조치를 시행한다"와 "전액 감액 처리한다"는 동일 내용이지만 GPT 정답과 표현이 달라 낮은 점수를 받을 수 있다. 이는 GPT로 생성한 ground truth에서 구조화 필드 이외의 자유 형식 요약 필드에 공통으로 발생하는 한계이며, 채점 점수는 "GPT 정답 표현과의 의미적 근사도"로 해석해야 한다.

## v1에서 v2로 전환한 이유

v1 채점 결과를 실제 모델 출력과 대조하는 과정에서 세 가지 구조적 문제가 확인됐다.

**1. 날짜 포맷 오채점**  
모든 프롬프트가 `YYYY-MM-DD` 포맷을 명시적으로 요구했지만, 모델이 `2024년 5월 18일` 형식으로 출력한 경우 GPT가 이를 내용 불일치로 처리했다. 날짜 값 자체는 맞는데 감점이 발생한 것이다.

**2. 필드 뭉개기**  
7개 필드를 GPT가 단일 점수로 평가했기 때문에 어느 필드가 실패했는지 알 수 없었다. 기관명 부서명 누락, 담당자 직책 누락, 제목 절삭이 모두 같은 점수 범위에 묻혔다.

**3. GPT rubric 모호**  
"핵심 정보 절반 정도 일치 = 0.5" 기준이 불명확해 GPT 재량이 컸다. 실제로 6/7 필드를 완벽히 맞춰도 0.5점이 나오는 케이스가 확인됐다.

## 채점 설계 과정에서 검토한 내용

### 프롬프트별 필드 요구 사항 확인

6개 프롬프트를 직접 확인한 결과, 모든 프롬프트가 동일하게 요구하고 있었다.

| 필드 | 요구 사항 |
|------|-----------|
| 기관명 | "발신 기관 또는 담당 부서명" — 부서명 포함 명시 |
| 담당자 | "담당자 이름 (직책 포함)" — 직책 포함 명시 (json_schema만 "포함 가능"으로 optional) |
| 작성일·처리기한 | "YYYY-MM-DD 형식으로 변환" — 포맷 변환 명시 |

이를 근거로 부서명 누락, 직책 누락은 모두 실제 추출 실패로 간주했다.

### 실제 모델 출력 불일치 패턴 분석

4500개 파일에서 필드별 불일치 패턴을 샘플링한 결과:

**기관명**: 모델이 부서명을 누락하고 기관명만 출력하는 패턴  
예) 정답: `행정안전부 청사시설과` → 출력: `행정안전부`  
→ 원문에 "행정안전부 청사시설과"가 명시되어 있으므로 추출 실패로 판정 가능

**문서번호**: 빈 문자열 출력, "문서 고유 번호"(설명 텍스트 출력), 한자 혼용, 글자 오삽입  
예) 정답: `강남교통-2024-1187` → 출력: `문서 고유 번호`  
→ 모두 명백한 오류, exact match로 충분

**작성일·처리기한**: 포맷만 다른 경우가 대부분  
예) 정답: `2024-05-18` → 출력: `2024년 5월 18일`  
→ 날짜 값은 같고 포맷만 다름

**담당자**: 이름은 맞고 직책만 누락하는 패턴이 일관됨  
예) 정답: `이서연 주무관` → 출력: `이서연`

**제목**: 끝 단어 절삭, 한자 혼용(학生·行政), 단어 누락 등 다양한 패턴

**핵심내용**: 원문 그대로 복사하거나 요약 품질이 낮은 경우

### 필드 분류 결정 과정

처음에는 기관명을 GPT로 처리하려 했으나, "프롬프트가 부서명 포함을 요구하는데 원문에도 있다면 부서명 누락은 GPT 판단이 필요 없는 명확한 오류"라는 판단으로 코드 처리로 결정했다.

담당자를 처음에는 "이름만 추출해서 코드 비교"하려 했으나, 동명이인 구별을 위해 직책이 의미 있고 직책 포함/누락에 따른 부분 점수 판단이 필요하다는 이유로 GPT 처리로 결정했다.

## 채점 기준 (확정)

### 코드 처리 필드 (4개)

**문서번호**
- 공백 정규화 후 exact match
- 일치 → 1.0 / 불일치·빈값 → 0.0

**작성일 / 처리기한**
- 한국어 날짜(`2024년 5월 18일`)와 ISO 날짜(`2024-05-18`) 모두 파싱 지원
- 날짜 값 + 포맷 모두 정확 → 1.0
- 날짜 값은 맞고 포맷만 틀림(한국어 형식 출력) → 0.8
- 날짜 값 틀림 / 파싱 불가 / 빈값 → 0.0

> **측정 범위 주의**: 문서 내 날짜는 모두 한국어 형식(`2024년 5월 18일`)으로 작성되어 있고, 정답은 ISO 형식(`2024-05-18`)을 요구한다. 따라서 이 필드의 점수는 **정보 추출 능력과 포맷 변환 능력을 동시에 측정**한다. 두 능력을 분리하려면 ISO 날짜가 원문에 직접 포함된 문서를 사용하는 별도 실험이 필요하다. 0.8 처리는 "값은 맞지만 포맷을 변환하지 못한" 경우를 부분 감점으로 구분하기 위한 설계이며, 포맷 변환 실패를 완전 오류와 동등하게 취급하지 않는다.

**기관명**
- 비교 전 정규화 적용: 괄호`()（）` 제거 후 공백 치환, 문자열 끝 조사(의·에서·으로·로·은·는·이·가) 제거, 공백 정규화
  - 예) "행정안전부의" → "행정안전부", "행정안전부(청사관리과)" → "행정안전부 청사관리과"
- 정답에 부서명이 없으면 exact match만으로 1.0 판정 (정답 자체가 기관명만인 경우)
- 정답에 부서명이 있을 때: 완전 일치 1.0 / 기관명만 일치(부서명 누락) 0.5 / 그 외 0.0

### GPT 처리 필드 (3개, 1회 호출로 묶음)

**담당자**
- 이름 + 직책 모두 일치 → 1.0
- 이름만 일치, 직책 누락 → 0.5
- 직책만 있고 이름 없음 / 완전 틀림 / 빈값 → 0.0

**제목**
- 핵심 단어 모두 포함, 동일 → 1.0
- 핵심 단어 대부분 있으나 일부 누락·변형 → 0.5
- 전혀 다름 / 빈값 → 0.0

**핵심내용** (점수: 1.0 / 0.5 / 0.0)

GPT가 다음 순서로 판단한다:
① 정답 핵심내용의 정보 항목을 나열
② 모델 출력에서 각 항목 포함 여부 확인 (표현이 달라도 내용이 같으면 포함)
③ 점수 결정:
- 1.0: 핵심 정보 항목이 모두 포함되고 간결하게 요약됨
- 0.5: 핵심 정보 항목 일부만 포함, 또는 원문을 그대로 복사
- 0.0: 핵심 정보 없음 / 완전 오류 / 빈값

원문 복사 판정: 정답 핵심내용의 핵심 문장이 3개 이상 그대로 반복되면 0.5 상한 적용.
GPT 출력이 0/0.5/1.0 이외의 값일 경우 코드에서 가장 가까운 허용값으로 클램프 처리.

> **주의**: 위 기준은 v3에서 적용될 개선안이다. **이 파일의 수치(eval_results.csv)는 구버전 루브릭 기준**으로 채점됐으며, 핵심내용 점수의 85.4%가 0.5에 집중되는 변별력 한계가 존재한다. v3에서 위 루브릭으로 재채점할 예정이다.

### schema_valid=False 샘플의 semantic_score 처리

`is_valid_json=True`이고 `schema_valid=False`인 샘플(필드 키 누락)도 field_scores를 계산한다. 누락 필드는 0.0으로 처리되어 semantic_score에 반영된다. 전체 4,500건 중 286건(6.4%)이 schema_valid=False다.

기법별 schema_valid=False 비율이 달라 semantic_score를 기법 간 직접 비교하면 편향이 발생한다. 아래는 전체 vs schema_valid=True 필터링 시 semantic_score 비교다.

| 기법 | 전체 semantic | schema_valid=True만 | schema_false 건수 |
|------|-------------:|-------------------:|------------------:|
| cot         | 0.9024 | 0.9025 |   1 |
| few_shot    | 0.9249 | 0.9264 |   8 |
| json_schema | 0.8860 | 0.8936 |  36 |
| structured  | 0.8094 | 0.8129 |   4 |
| xml         | 0.8196 | **0.8545** | 160 |
| zero_shot   | 0.8611 | 0.8926 |  77 |

xml은 schema_false가 160건(21.3%)으로 가장 많고, 필터링 시 semantic_score가 +0.035 상승해 기법 간 순위에 영향을 준다. 기법 비교 분석 시에는 schema_valid=True 샘플만 대상으로 하거나 schema_false 비율을 함께 보고해야 한다.

## 채점 실행 중 발생한 버그

### 버그 1: `_extract_json`이 list 반환

**발생**: 4500개 중 407개 처리 후 크래시  
**에러**: `AttributeError: 'list' object has no attribute 'get'`  
**원인**: 일부 모델 출력이 JSON 배열(`[...]`)로 파싱되어 list가 반환됨. `_extract_json`의 최상위 `json.loads()` 경로에 dict 타입 검증이 없었음  
**수정**: `json.loads()` 결과가 dict인 경우만 반환하도록 수정

```python
# 수정 전
try:
    return json.loads(text)
except json.JSONDecodeError:
    pass

# 수정 후
try:
    obj = json.loads(text)
    if isinstance(obj, dict):
        return obj
except json.JSONDecodeError:
    pass
```

### 버그 2: 필드 값이 list인 경우

**발생**: 버그 1 수정 후 재실행, 1434개 처리 후 크래시  
**에러**: `AttributeError: 'list' object has no attribute 'strip'`  
**원인**: parsed dict는 올바르게 가져왔으나, dict 내부 필드 값이 string이 아닌 list인 케이스 존재. 예) `{"기관명": ["행정안전부", "청사관리과"]}`  
**수정**: 채점 함수 호출 전에 parsed dict의 모든 필드 값을 string으로 정규화

```python
p = {
    k: v if isinstance(v, str) else ""
    for k, v in (parsed or {}).items()
}
```

### 버그 3: 한국어 날짜 파싱 실패

**발생**: 테스트 실행 후 `score_작성일`, `score_처리기한`이 전부 0.0으로 나옴  
**원인**: `dateutil.parser`가 `2024년 5월 18일` 형식을 파싱하지 못함  
**수정**: 정규식으로 한국어 날짜 형식을 직접 파싱하는 `_parse_date` 함수 추가

```python

    m = re.fullmatch(r"(\d{4})-(\d{2})-(\d{2})", s)

    # YYYY년 M월 D일
    m = re.fullmatch(r"(\d{4})년\s*(\d{1,2})월\s*(\d{1,2})일", s)
    if m: ...
```

| 1.5b | 0.5545 | 0.8013 |
| 3b   | 0.5597 | 0.8840 |
| 7b   | 0.5818 | 0.9151 |

v1의 flat한 결과는 날짜 포맷 오채점 등 채점 편향이 전 모델에 동일하게 작용한 결과로 보인다.

**해석 주의**: 세 모델은 파라미터 수(1.5B/3B/7B) 외에 학습 데이터 규모, RLHF 방식, 아키텍처 세부사항이 통제되지 않았다. 따라서 관찰된 성능 차이를 "모델 크기 효과"로 단정할 수 없으며, "qwen2.5 시리즈 내에서 파라미터 수가 많은 모델일수록 관찰된 성능 차이"로 기술하는 것이 정확하다.

### 2. 세 메트릭의 분리 해석

`is_valid_json`, `schema_valid`, `semantic_score`는 서로 다른 능력을 측정하므로 독립적으로 해석해야 한다.

| 메트릭 | 측정 대상 | 비고 |
|--------|-----------|------|
| `is_valid_json` | JSON 파싱 가능 여부 | 구조적 유효성 — 내용과 무관 |
| `schema_valid` | 7개 필드 키 존재 여부 | 스키마 준수 — 값 정확도와 무관 |
| `semantic_score` | 필드 값의 의미적 정확도 | 내용 품질 — 구조 유효성과 독립 |

is_valid_json이 높아도 semantic_score가 낮을 수 있다. 실제로 전체 샘플의 is_valid_json 평균은 0.994이지만 semantic_score 평균은 0.867로 12.7%p 차이가 난다. 세 지표를 단일 지표로 환원하면 이 차이가 숨겨진다.

### 3. semantic_score 천장 효과의 구조적 원인

semantic_score의 56.8%(2,557/4,500)가 0.9286이라는 단일 값에 집중된다. 이는 채점 루브릭의 한계가 아닌 **필드 구조 자체의 문제**다.

- 핵심내용을 제외한 6개 필드의 평균 점수: 0.92 이상 (대부분 {0, 0.5, 1.0}의 이산값)
- 핵심내용 점수: 0.40~0.50 수렴 (v2 루브릭의 변별력 한계)
- 결과: 7개 필드 평균이 "핵심내용=0.5 + 나머지=1.0" 조합에 집중 → (6×1.0 + 0.5)/7 = 0.9286

천장 효과의 완전한 해소는 3.1에서 설계한 핵심내용 루브릭 개선(v3 재채점)으로 가능하다.

### 4. 핵심내용이 전체 semantic_score의 병목

모든 모델·기법에서 `score_핵심내용`이 0.40~0.50 수준으로 가장 낮다. 나머지 6개 필드는 대부분 0.85 이상인데 핵심내용만 절반 수준이다. 요약 품질이 실질적인 성능 병목임.

### 3. structured 기법의 기관명 부서명 누락 패턴

structured 기법에서 **모델과 무관하게** 기관명 부서명 누락률이 40% 이상으로 나타났다.

| 기법 | 1.5b | 3b | 7b |
|------|---:|---:|---:|
| structured | 45.4% | 40.0% | 41.1% |
| few_shot   | 0.0%  | 0.0%  | 0.0%  |
| cot        | 10.0% | 0.4%  | 2.0%  |

프롬프트의 `Step 1. 발신 기관 또는 담당 부서를 확인한다` 지시가 모델로 하여금 기관명만 추출하고 부서명을 별도로 인식하게 만드는 것으로 추정된다.

### 5. qwen2.5:3b + structured 조합의 문서번호 빈값

qwen2.5:3b에서 structured 기법으로 실험 시 문서번호 빈값 출력이 47.2%로 나타났다. 1.5b(2.0%), 7b(0.8%)와 뚜렷하게 다르다.

**카이제곱 검정 결과** (Bonferroni α = 0.0167):

| 비교 | chi2 | p | 유의 |
|------|-----:|---:|------|
| 1.5b vs 3b | 135.26 | <0.001 | * |
| 1.5b vs 7b |   0.58 | 0.446  | ns |
| 3b vs 7b   | 145.01 | <0.001 | * |

3b가 1.5b·7b와 통계적으로 유의하게 다르고(p<0.001), 1.5b와 7b 간 차이는 유의하지 않다. **3b 특유 패턴**으로 확인된다.

원인은 structured 프롬프트의 "없으면 빈 문자열로 표기한다" 지시와 3b 모델 간 상호작용으로 제한적으로 해석한다. 동일 지시에 1.5b·7b는 영향을 받지 않으므로, 3b의 instruction-following 특성과 관련된 것으로 추정되나 현 실험 설계로는 원인을 특정할 수 없다.

### 5. few_shot이 날짜·담당자 포맷 준수율 최고

few_shot 기법에서 `score_작성일 = 1.0`, `score_담당자 = 1.0`으로 전 기법 중 유일하게 완벽하다. 예시(few-shot example)가 날짜 포맷과 직책 표기 방식을 직접 보여주는 효과로 해석된다.

## 실험 환경 한계

### 재현성

모든 추론은 temperature=0.0(greedy decoding)으로 고정했다. 이론적으로는 동일 입력에 대해 동일 출력이 나와야 하지만, 실제로는 완전한 재현성이 보장되지 않는다. GPU 병렬 연산의 부동소수점 누적 오차, llama.cpp 내부의 연산 순서, Ollama 버전, 메모리 상태에 따라 동일 입력에서도 출력 토큰이 일부 달라질 수 있다. 따라서 temperature=0.0은 "재현성에 근접한 조건"이지 "완전한 재현성 보장"이 아니다.

### 실험 실행 이력

본 실험은 3회에 걸쳐 점진적으로 완성됐다.

| 실행 | 일시 | runs_per_cell | 신규 생성 | 스킵 |
|------|------|---:|---:|---:|
| 1차 | 2026-05-18 14:58 | 20 | 1,708 | 92 |
| 2차 | 2026-05-18 17:21 | 50 | 2,160 | 1,800 |
| 3차 | 2026-05-19 08:38 | 50 | 540 | 3,960 |

1차 실행 시 runs_per_cell=20으로 index 0~19를 먼저 생성하고, 이후 50으로 늘려 index 20~49를 추가했다. 각 실행은 기존 파일을 덮어쓰지 않으므로 파일 간 혼재는 없다. 단, 1차(2026-05-18)와 3차(2026-05-19) 사이 약 18시간의 시간 차가 존재하며, 앞서 언급한 GPU 비결정성 요인으로 인해 실행 시점 간 극소수 출력 차이가 발생했을 가능성은 배제할 수 없다.

## GPT 채점 반복 일관성 검증

**방법**: 무작위 50개 샘플(seed=42)을 동일 프롬프트·temperature=0.0으로 3회 반복 채점. 샘플당 3회 점수의 표준편차를 계산 (총 150회 API 호출).

**결과**:

| 필드 | mean_std | max_std | 변동 발생 |
|------|--------:|--------:|--------:|
| 담당자 | 0.0000 | 0.0000 | 0 / 50 |
| 제목 | 0.0000 | 0.0000 | 0 / 50 |
| 핵심내용 | 0.0058 | 0.2887 | 1 / 50 |

담당자·제목은 50개 전 샘플에서 완전히 일관됐다. 핵심내용은 1건(2%)에서 점수 변동이 발생했으며 해당 샘플의 std=0.2887 (0/0.5/1.0 척도에서 한 단계 차이). 전체 150건 중 변동은 1건(0.7%)이다.

**해석**: mean_std < 0.1 기준으로 GPT 채점의 변별력 주장을 제한할 수준은 아니다. 단, 핵심내용 필드에서 2% 수준의 산발적 불일치가 존재하므로 해당 필드의 경계값(0.5 vs 1.0) 판단에는 오차 가능성이 있다.

## 통계 검정 결과 (기법 간 쌍별 비교)

**검정 방법**
- `is_valid_json` / `schema_valid` (이진값): 카이제곱 검정

### 패턴 유형 분류 (collapse_threshold.csv: json_pattern / schema_pattern)

`_first_degradation` 함수는 단조 감소 패턴을 가정하므로, 역방향 패턴(초반 낮고 후반 높음)에서는 잘못된 해석을 낳을 수 있다. 이를 보완하기 위해 budget별 성공률의 첫값·끝값을 비교해 패턴 유형을 추가로 기록한다.

| 유형 | 정의 | 예시 |
|------|------|------|
| `collapse` | 첫 budget 성공률 > 마지막 budget 성공률 | 길어질수록 안정성 저하 |
| `recovery` | 첫 budget 성공률 < 마지막 budget 성공률 | 짧은 문서에서 오히려 불안정 |
| `stable` | 첫·끝 성공률 동일 | 전 구간 안정 |

**주요 관찰**: qwen2.5:1.5b + xml의 schema_pattern=**recovery**. budget_200에서 schema_valid=0.00이고 budget_4000에서 0.62로 오히려 상승한다. 이는 xml 포맷 특유의 패턴으로, 짧은 문서에서 1.5b 모델이 핵심내용 필드를 생략하는 현상(budget_200 전 샘플에서 핵심내용 키 누락 확인)에 기인한다. 단조 감소를 가정한 `_first_degradation`은 이 케이스에서 "200tok부터 저하"로 보고하지만, 실제로는 길이가 증가할수록 안정성이 회복되는 패턴이다.

**분류 한계**: 첫값·끝값만 비교하므로 중간 구간의 비단조 변동은 반영되지 않는다. 샘플 수(N=50/구간)에 따른 통계적 변동으로 패턴이 오분류될 수 있다.
